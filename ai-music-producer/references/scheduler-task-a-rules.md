## ⛔ 流派/主题去重（强制执行，Phase 1 选题前必走）

> **参考教训**：6/6 06:03 A 任务「回声」(工业电子 #16) 与 6/5「大妈的脸」(工业电子 #16) 撞车，3 首产出中 1 首作废。问题根因：标题写"7天去重"但下面没有 ⛔ 强制块，agent 跳过。

### 去重时间窗

| 窗口 | 范围 | 维度 |
|------|------|------|
| **7 天** | `time_filter.start = <今天-7天>` | 流派 + 主题 + **歌名** |
| **24 小时** | `time_filter.start = <昨天>` | 流派 + 主题 + **歌名** |

### 执行步骤（必须按序）

1. `search_notes(query="🎵 作品档案", time_filter={start:"<今天-7天>"})` → 收 7 天内已用
2. `search_notes(query="🎵 作品档案", time_filter={start:"<昨天>"})` → 收 24 小时内已用
3. 解析每个档案的 **「歌名/标题」** + 「流派：」字段（标题或正文首行）+「主题来源」字段，构建已用集合
4. **流派去重**：本次所选流派不得与 7 天内已用流派重复（近似流派也需规避，如本周用了 Dream Pop 就不再选 Dream Pop / 梦幻流行）；**主题池 = 知识库候选 − (7天已用 ∪ 24h已用)**
5. 可用流派不足 5 种 → 可自由扩展新流派补足（参考池以外亦可）
6. 从可用/新扩展流派中为本次 5 首各分配一个（流派, 主题）对

### ⛔ 歌名去重（Phase 1 确定歌名后强制检查）

> **参考教训**：6/11 A 任务选了「喷泉」「经纬」，但这两首 6/10 已创作过（只是流派不同），导致 6/11 实际只有 1 首新歌。7 天窗口过期后同名歌曲会**覆盖**旧文件目录。

确定 3 首歌名后、进入 Phase 2 之前，执行**两层去重**：

**第一层：全历史去重（读 songs.json，最高优先级）**
1. 用 `run_python` 读取 songs.json，**只提取 title 列表并输出为一行逗号分隔字符串**（不要把完整 JSON 输出到上下文）
2. 示例代码：
```python
import json, os
path = os.path.expanduser("~/Library/Application Support/remio/Users/F2313D5DDFE8FCF316DC1149F06BB14B/agent/music-vault/data/songs.json")
with open(path) as f:
    data = json.load(f)
titles = [s.get('title', '') for s in data['songs']]
print(f"全量已用歌名 {len(titles)} 首:")
print('、'.join(titles))  # 只输出歌名，不输出完整 JSON
```
3. 将 3 个候选歌名逐一对比输出的全量歌名列表

**第二层：7 天近期去重（search_notes，二次确认）**
4. 从 Step 1-2 搜索结果中提取所有 7 天内已用的 **歌名**（标题中 `·` 或 `—` 前的部分，或笔记标题中 `🎵` 后的部分）
5. 合并全量集合 ∪ 7 天集合 = **完整黑名单**
6. 候选歌名 vs 完整黑名单

**判定与处理**：
7. **⚠️ 重复判定**：歌名完全相同（忽略大小写/空格），或歌名高度相似（如「喷泉」vs「小喷泉」），都算重复
8. 重复的歌名必须**重新命名或换主题**，不得沿用
9. 在去重报告中列出全量歌名数和 7 天歌名列表

## 结构多样性（不套同一模板）

## 歌词创作规则

### ⛔ 路径选择（强制路径 B）

**固定使用路径 B（全手写），禁止使用 MiniMax 歌词 API（路径 A）。**

原因：MiniMax 歌词 API 返回的骨架经诗化审核后，GLM 模型会极端压缩行长（2-4字碎片化），破坏叙事性。路径 B 用 DeepSeek 全手写，行长分布更自然（5-10字为主），叙事完整。

### 路径 B：全手写歌词（默认路径）

用 **SiliconFlow** (`nex-agi/Nex-N2-Pro`) 创作完整歌词；SiliconFlow 不可用则 fallback DeepSeek。遵循以下结构模板：

```
请创作一首完整的歌词，严格遵循以下结构和行数要求：
[Intro] - 2-3行氛围铺陈
[Verse 1] - 6-8行叙事
[Pre-Chorus] - 2-3行情绪过渡
[Chorus] - 4-6行核心 Hook
[Verse 2] - 6-8行深化
[Pre-Chorus] - 2-3行
[Chorus] - 4-6行（递进变化）
[Instrumental] - 标记纯器乐段落
[Verse 3] - 6-8行新视角
[Chorus] - 4-6行（第三次变化）
[Bridge] - 4-6行转折
[Solo] - 标记纯器乐段落
[Chorus] - 4-6行终版
[Outro] - 3-4行收束
总计纯歌词行数要求：≥60行（不含结构标签行）
每个段落之间留 1 行空行。
歌词风格：用画面和意象说话，禁止直白表达情感。
```

### 诗化审核（第二环）

**⛔ 必须使用 DeepSeek 模型执行诗化审核，禁止使用 GLM。**

DeepSeek 对「短句」规则的执行更克制，不会过度压缩行长。GLM 会把正常句子极端压缩成 2-4 字碎片。SiliconFlow 不做诗化审核（它是创作用的）。

作为「词境审核师」审核整首歌词。规则：

**【词境之场】**：场有双力——根力向情感扎根，流力向旋律流淌。不直说只暗示，说出三分留七分。宁可短句不可长句，宁可空白不可填满，宁可跳跃不可平铺。

**【抽象之梯下沉】**：每句歌词在梯子底部——用物件/场景/动作说话。禁止「想念」「孤独」「快乐」「悲伤」「爱」「迷茫」等情感词。

**【诗歌四要素】**：
1. 行顿：每4行必有≤3字短句/省略号/空行（换气口）
2. 酌字：每行≤10字，不用虚词
3. 跳跃：允许意象非线性跳跃
4. 留白：不说透，让听众续写

**【五感选材】**：优先视觉+听觉意象，偶尔通感跳跃，但每个意象只取一个切面，不堆叠。

**【Chorus递进】**：每次副歌改1-2行（提出→加深→升华）。禁止说理句。

### lyric-qa 质检（第三环）
调用 `lyric-qa` skill 四维度检测。画面感<7分或陈词滥调>3处则回炉（最多2轮）。通过后调用 `music-prompt-templates` skill 输出双平台提示词。

### 歌词格式铁律
- 只允许 `[Tag]` + 纯歌词，禁止括号描述词
- 禁止发音黑名单：画框、缝补、依归、干涸、禁止项、避让

## 时长铁律（目标 3:00-4:00）
- 纯歌词≥55行，目标60-75行（不含 `[Tag]` 行）
- 实测：25-30行≈1:30❌，55行≈2:30及格，60-70行≈3:00-3:30✅
- 至少2个纯器乐段落（`[Instrumental]`/`[Solo]`）
- **⛔ 完成后必须数行数，N<55立即补充，N<50判定失败**
- **⛔ 每首必须输出**：`✅ 歌词行数检查：纯歌词 N 行（目标 ≥55），预估时长 X:XX`
- **⛔ 绝对禁止<50行**

## 目录命名铁律（Phase 2 输出时强制）

⛔ 音乐文件目录必须使用日期前缀格式：`YYYY-MM-DD_歌名/`

**示例：**
```
~/Music/音乐项目/2026-06-14_索引/索引_v1.mp3
```

**规则：**
- 日期 = Task A 执行当天
- 歌名 = 纯歌名（不含日期、不含标点）
- 目录内文件仍用纯歌名命名：`歌名_v1.mp3`、`歌名_lyrics.txt`
- ⛔ 禁止使用无前缀的纯歌名目录（会触发覆盖 bug）

## pending_audio.json 结构（Phase 3，Task A 输出时写入）

Task A 结束时，更新 `music-vault/data/pending_audio.json`：
```json
{
  "date": "2026-06-14",
  "note_ids": ["noteId1", "noteId2"],
  "songs": [
    {"title": "歌名", "song_dir": "~/Music/音乐项目/2026-06-14_歌名"}
  ]
}
```
Task A' 读取 `songs[].song_dir` 作为输出目录，不再自行拼接路径。

## 产出校验（任务结束前必须执行）
1. 调用 `search_notes` 搜索今天创建的 `作品档案`（`time_filter` 设置今天，`limit: 5`）
2. 确认今天有 ≥1 个新作品档案
3. 如果 0 个 → 输出 `❌ TASK VALIDATION FAILED: 0 songs created today`（这会让调度器标记该次运行为异常）
4. 报告格式：`创作成功N首/失败M首/补集修复K首`
